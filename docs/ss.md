- 「COM7のUSBを抜いて外部電池（乾電池）だけで動かしたい」→ USB接続時は動くが、電池のみだと後輪左サーボが回転しない事象が発生

### 指先用IMU②の追加と配線
- 手首IMU①（0x68）に加え、指先用に2個目のMPU-9250②を追加する計画
  - 同一I2Cバス（D14/D15）を共用し、AD0ピンでアドレスを0x69に分離する方針を確認
- ②を接続後、①②とも全く応答しない（WHO_AM_I=0xFF、I2Cスキャナーでも0件）という不具合が発生
  - 当初「①も同時接続していた」という前提で対策を検討していたが、実際は「②のみを接続していた」ことが後に判明し、切り分けの前提を修正
  - テスター（導通・電圧）で配線に異常なしを確認済みとの申告あり
  - プルアップ抵抗の追加、I2Cクロックを100kHz→10kHzに変更、いずれも改善なし
  - コミュニティ情報（Discord共有）よりCS・FSYNCピンの配線が必要と判明（CS→3.3VでI2Cモード固定、FSYNC→GND）→ 配線を追加するも改善なし
  - 外部から共有されたコード2種（ダミー乱数生成コード、Arduino Bridge依存コード）は本件の解決に無関係と判定し却下
- `I2C_Deep_Diag`を作成し、SPRESENSEコアの`Wire.cpp`実装（`endTransmission`の返り値：0=ACK, 2=NACK, 4=バス異常）を根拠にした診断を実施
  - 返り値は一貫して2（NACK）→ 当初「バスは正常、デバイスのみ無応答＝モジュール不良」と結論
- オシロスコープ実測により診断が確定
  - SCL：約4.7Vでパルス正常に観測
  - SDA：**約0.4Vに張り付き**、駆動しても0.1Vpp程度しか動かない
  - 9クロック法によるバスロック解除も無効
  - → SDAパッドの物理破損（クランプ電位の典型値）と結論。NACK＝バス正常という前段の解釈より、電気的破損の方が実態に近いと判断
- 破損原因の推定：**SPRESENSE拡張ボードのI/O電圧ジャンパが5V設定になっていた**。MPU-9250の絶対最大定格（約3.6V）を超過しており、これが指先用IMU②破損の最有力原因
- ジャンパー線を押さえるとZ軸の値だけ暴れる（X/Yは安定）現象を確認 → チップ自体は健全、配線（ジャンパー線かブレッドボード内部）の接触不良が原因と最終確定
- **結論：IMU②（偽造/リマークMPU-9250）はSDAパッド破損により交換不可・使用不可。交換部品としてMPU-6050系（レジスタ互換）を推奨したが、後日ジャイロのみの方針に転換（下記07-09参照）**
- `docs/2026-07-06_作業報告.md`に診断経緯を記録、READMEも進捗・残タスク・電源構成・3.3V注意書きを更新してプッシュ

## 2026-07-09（本日）：ジャイロ単体方式への転換〜ESP32単体制御への統合

### 「加速度不要・ジャイロのみ」への設計転換
- 指先の役割は「指をさして回転→その方向へ曲がる」という**回転速度ベースの操作**であり、絶対角度（加速度による重力方向の傾き検出）は不要と判断
- ジャイロの角速度（dps）をそのまま旋回量に変換する設計を採用（曲げセンサーの`flatValue`と同じ「起動時ゼロ点補正」の考え方を踏襲）
- 手持ちの代替センサーとして**L3GD20（ST社製3軸ジャイロ）**と**圧電振動ジャイロ（アナログ・1軸）**を比較
  - 圧電ジャイロはI2Cを使わないためIMU②で発生したような電圧トラブルのリスクがなく、当初はこちらを推奨
  - 最終的にL3GD20（3.3V動作品）を採用し、配線・実装を開始

### L3GD20の配線とコード作成
- L3GD20はCSピンでSPI/I2Cを切り替える設計（**CS→3.3VでI2Cモード固定**、今回のMPU-9250のCS/FSYNCの教訓がそのまま活きた）、SA0でアドレス0x6A/0x6Bを選択する仕様と確認
- `L3GD20_Gyro_Test.ino`を新規作成
  - I2Cアドレス自動判定（0x6A/0x6B両対応）、WHO_AM_I確認（0xD4/0xD7両対応）
  - ST社特有の実装注意点に対応：複数バイト読み出し時はレジスタ番地の最上位ビットを立てる（自動インクリメント指示）、出力はリトルエンディアン
  - 起動時の静止状態でのゼロ点補正機能を実装

### L3GD20の接続不良トラブルシュート
- 初回テストで実際のジャイロ値取得に成功（ゼロ点補正値 X/Y/Z が現実的な小さい値）も、間欠的に「Failed to read from i2c」が発生
  - 1回の失敗で即座に再検出する設計だと過敏なため、**5回連続失敗するまで粘る**仕様に変更
- 再接続のたびにゼロ点補正をやり直す設計により、配線をいじっている最中（=センサーが動いている最中）に補正が走り、オフセットが乱れる不具合を確認
  - **電源投入後の最初の1回だけ補正し、以降は使い回す**設計に修正（`calibrated`フラグ追加）
- ジャンパー線を押さえるとZ軸の値だけ大きく暴れる（X/Yは安定）現象を確認 → **センサー本体ではなく配線（ジャンパー線 or ブレッドボード内部接点）の問題**と結論。センサーは健全と確定

### ESP32単体化（曲げ＋ジャイロ＋ESP-NOW）への設計統合
- 「ESP32が送信するなら、センサー類もESP32に直結すべきでは」という方針転換
  - L3GD20をESP32へ配線：SDA→GPIO21、SCL→GPIO22（Wireライブラリのデフォルト）、CS→3.3V、SA0→GND/3.3V
  - 曲げセンサーもESP32のADC1系統（**GPIO34**、Wi-Fi使用中のADC2系統は不安定なため回避）に接続する方針を確認
  - 曲げセンサーの分圧回路は、SPRESENSEのIOREF(1.8V)基準からESP32の3.3V基準に組み替える必要があり、`BEND_MAX_DIFF`等の再校正が必要と申し送り
  - この統合により**SPRESENSE↔ESP32のUARTブリッジ（GPIO16/17）は車体制御チェーンにおいて不要**に。SPRESENSEはSense_Core（カメラ・超音波・振動・手首IMU）用途として引き続き独立稼働
- `Bend_Gyro_ESP_Sender.ino`を新規作成し、以下を1本に統合
  - 曲げセンサー→スロットル（`Bridge_SPRESENSE`のロジックを流用）
  - L3GD20→ステア（`L3GD20_Gyro_Test`の実装をそのまま流用）
  - ESP-NOW送信（`Bridge_ESP_Sender`のDriveCmd構造体・ブロードキャスト方式を流用）
  - 合成ロジック：`left = throttle - max(0, steer)` / `right = throttle - max(0, -steer)`（既存の「片側を止めて曲がる」思想を段階的にしたもの）

  - 対処：ENボタンでの手動リセット、あるいはUSB抜き差し
  - 恒久対策として「ENピン-GND間に10µFコンデンサを追加」というハード側の定番対策、および「VS Code内蔵モニターではなく`arduino-cli monitor`を使う」というツール側の回避策を提示（**この問題はスケッチのコードでは解決不可能**と明言）

  - ジャイロ由来のステア値を**スルーレート制限**（`STEER_STEP`、1ループあたりの最大変化量）で滑らかに追従するよう変更

### ESP32電源ピンの整理（VN と 5V の混同）

- 「乾電池の本数は電圧か容量か」という質問 → 直列は電圧が加算、並列は容量が加算されると説明
- ESP32のVIN(5V)ピンの入力電圧耐性をWeb検索で確認：レギュレータの理論上限は約20Vだが、**実用上は6〜7.4V程度が推奨**（発熱のため）
- 「ESPの5Vは？」「じゃあ5VとVNと電池のVINを繋ぐわ」との発言に対し、**強く制止**
  - 「VN」はESP32チップのGPIO39（ADC専用ピン）を指す可能性が高く、電池の6Vを直接繋ぐとADC入力の耐圧（3.6V程度）を超えて損傷するリスクがあると説明
  - 正しくは「5V（=VIN）」だけに電池+を繋ぎ、「VN」は未接続のままにすることで合意
- 車体側の電源に関するやり取りで、以前の「USBなしで電池単体だと基板が発熱していた」という報告と「LEDが点灯しなかった」という報告が矛盾するように見えたため一度指摘したが、ユーザーから「両方とも同じ現象（ブラウンアウトループ）の一部」と訂正が入り、**指摘の仕方が悪かった点を謝罪**
  - 教訓：一度立てた解釈に固執せず、早い段階で柔軟に見直すべきという フィードバックを受けた

### 一般知識系の質問（本プロジェクトに関連する文脈で）

- 「COM番号はパソコンによって異なるか」→ Windowsが機器接続履歴に基づきPCごとに割り当てるため異なる、と回答。チームメンバーが別PCで書き込む際は毎回確認が必要と申し送り
- 「ESPは今Bluetooth通信だったよな？」→ **訂正**：実際はESP-NOW（Wi-Fi電波を使うがペアリング不要の独自プロトコル、最大250バイト/回）であり、Bluetoothではないと説明

### AR中継アーキテクチャの具体化

- 「送信側で車体だけでなく、SPRESENSE側かサーバーにも送りたい。コントローラーの値をスマホに無線反映し、ARに反映したものを動かしたい」との要望
  - ESP-NOWはESP32同士でしか届かないためスマホには直接届かない点を説明
  - ブロードキャスト方式のため、**送信側のコードを変更せずに中継用ESP32を追加するだけで**車体と同時受信できる利点を確認
  - 経路案：送信ESP32 → (車体 / 中継ESP32) → 中継ESP32はUSBでPCに接続 → `Dashboard/bridge.py`拡張 → スマホのブラウザ（WebAR）
  - SPRESENSEにWi-Fi機能があるかを確認したところ「ある」との回答

### SPRESENSE LTE拡張ボード＋SIMカードの検討

- SPRESENSEのWi-Fiに加え、「後にSIMカードと繋いでいつでも使えるようにしたい」との構想が判明
  - 当初は費用・実装の複雑さを理由に大会前の対応は非推奨と回答
  - 「SIMカードは既に提供してもらっている」「LTE拡張ボードも頼めば提供してもらえる」と判明し、費用面の懸念は解消
  - ただし実装・検証にかかる時間は依然として課題と結論。**Wi-Fi構成を本線として先に完成させ、LTEは並行の挑戦・保険を確保した上での追加**という優先順位で合意
  - ソニー純正の拡張ボードで公式ライブラリがあるため、偽造品リスクはないが、APN設定など固有の設定が必要になる点を申し送り

## AR/VR拡張構想（README追記内容の要約・その後の具体化含む）

- 曲げ＋ジャイロの`left/right`信号はハード非依存のため、物理ラジコンだけでなくUnity上の3Dオブジェクト操作にも転用可能という構想
- 技術構成案：ESP-NOW→受信ESP32→PC→Python中継サーバー（bridge.py拡張）→ブラウザ（WebXR/AR.js）
- Tier A（AR：スマホカメラに仮想カー重畳）→ Tier B（VR：フルCG空間）→ Tier C（Gemini物体認識との連携）の順で段階的に着手する方針
- 別途、チームメンバー向けに以下を口頭伝達用にまとめて提供
  - センサー値を3Dオブジェクトの動きに反映する必要があること
  - Unityで動く乗り物/生き物のモデルが必要なこと
  - まずAR/VR環境の構築を優先すること
  - モデルはテスト用にネット上の既存アニメーション付きモデルを使用し、本番用は可能であればBlenderで自作する方針
- **その後、具体的な中継経路として「送信ESP32はそのまま（ブロードキャストのため変更不要）→中継用ESP32を追加→USBでPC→bridge.py拡張→スマホWebAR」という実装案まで整理**
- SPRESENSEのWi-Fi機能（保有済み）に加え、将来的にLTE拡張ボード＋SIMカード（共に提供依頼可能）でオフライン/どこでも動作を目指す構想も浮上
  - 大会本番はWi-Fi構成を本線とし、LTEは並行して余力があれば挑戦する位置づけ
- **現時点では構想段階であり、実装は未着手**
